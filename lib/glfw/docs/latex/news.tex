\hypertarget{news_features_34}{}\doxysubsection{New features in version 3.\+4}\label{news_features_34}
\hypertarget{news_runtime_platform_34}{}\doxysubsubsection{Runtime platform selection}\label{news_runtime_platform_34}
GLFW now supports being compiled for multiple backends and selecting between them at runtime with the \mbox{\hyperlink{intro_guide_GLFW_PLATFORM}{GLFW\+\_\+\+PLATFORM}} init hint. After initialization the selected platform can be queried with glfw\+Get\+Platform. You can check if support for a given platform is compiled in with glfw\+Platform\+Supported.\hypertarget{news_standard_cursors_34}{}\doxysubsubsection{More standard cursors}\label{news_standard_cursors_34}
GLFW now provides the standard cursor shapes GLFW\+\_\+\+RESIZE\+\_\+\+NWSE\+\_\+\+CURSOR and GLFW\+\_\+\+RESIZE\+\_\+\+NESW\+\_\+\+CURSOR for diagonal resizing, GLFW\+\_\+\+RESIZE\+\_\+\+ALL\+\_\+\+CURSOR for omnidirectional resizing and GLFW\+\_\+\+NOT\+\_\+\+ALLOWED\+\_\+\+CURSOR for showing an action is not allowed.

Unlike the original set, these shapes may not be available everywhere and creation will then fail with the new GLFW\+\_\+\+CURSOR\+\_\+\+UNAVAILABLE error.

The cursors for horizontal and vertical resizing are now referred to as GLFW\+\_\+\+RESIZE\+\_\+\+EW\+\_\+\+CURSOR and GLFW\+\_\+\+RESIZE\+\_\+\+NS\+\_\+\+CURSOR, and the pointing hand cursor is now referred to as GLFW\+\_\+\+POINTING\+\_\+\+HAND\+\_\+\+CURSOR. The older names are still available.

For more information see \mbox{\hyperlink{input_guide_cursor_standard}{Standard cursor creation}}.\hypertarget{news_mouse_passthrough_34}{}\doxysubsubsection{Mouse event passthrough}\label{news_mouse_passthrough_34}
GLFW now provides the \mbox{\hyperlink{window_guide_GLFW_MOUSE_PASSTHROUGH_hint}{GLFW\+\_\+\+MOUSE\+\_\+\+PASSTHROUGH}} window hint for making a window transparent to mouse input, lettings events pass to whatever window is behind it. This can also be changed after window creation with the matching \mbox{\hyperlink{window_guide_GLFW_MOUSE_PASSTHROUGH_attrib}{window attribute}}.\hypertarget{news_wayland_app_id_34}{}\doxysubsubsection{Wayland app\+\_\+id specification}\label{news_wayland_app_id_34}
GLFW now supports specifying the app\+\_\+id for a Wayland window using the \mbox{\hyperlink{window_guide_GLFW_WAYLAND_APP_ID_hint}{GLFW\+\_\+\+WAYLAND\+\_\+\+APP\+\_\+\+ID}} window hint string.\hypertarget{news_features_34_angle_backend}{}\doxysubsubsection{Support for ANGLE rendering backend selection}\label{news_features_34_angle_backend}
GLFW now provides the \mbox{\hyperlink{intro_guide_GLFW_ANGLE_PLATFORM_TYPE_hint}{GLFW\+\_\+\+ANGLE\+\_\+\+PLATFORM\+\_\+\+TYPE}} init hint for requesting a specific rendering backend when using \href{https://chromium.googlesource.com/angle/angle/}{\texttt{ ANGLE}} to create Open\+GL ES contexts.\hypertarget{news_captured_cursor_34}{}\doxysubsubsection{Captured cursor mode}\label{news_captured_cursor_34}
GLFW now supports confining the cursor to the window content area with the GLFW\+\_\+\+CURSOR\+\_\+\+CAPTURED cursor mode.

For more information see \mbox{\hyperlink{input_guide_cursor_mode}{Cursor mode}}.\hypertarget{news_features_34_init_allocator}{}\doxysubsubsection{Support for custom memory allocator}\label{news_features_34_init_allocator}
GLFW now supports plugging a custom memory allocator at initialization with glfw\+Init\+Allocator. The allocator is a struct of type GLFWallocator with function pointers corresponding to the standard library functions {\ttfamily malloc}, {\ttfamily realloc} and {\ttfamily free}.

For more information see \mbox{\hyperlink{intro_guide_init_allocator}{Custom heap memory allocator}}.\hypertarget{news_features_34_position_hint}{}\doxysubsubsection{Window hints for initial position}\label{news_features_34_position_hint}
GLFW now provides the \mbox{\hyperlink{window_guide_GLFW_POSITION_X}{GLFW\+\_\+\+POSITION\+\_\+X}} and \mbox{\hyperlink{window_guide_GLFW_POSITION_Y}{GLFW\+\_\+\+POSITION\+\_\+Y}} window hints for specifying the initial position of the window. This removes the need to create a hidden window, move it and then show it. The default value of these hints is {\ttfamily GLFW\+\_\+\+ANY\+\_\+\+POSITION}, which selects the previous behavior.\hypertarget{news_features_34_win32_keymenu}{}\doxysubsubsection{Support for keyboard access to Windows window menu}\label{news_features_34_win32_keymenu}
GLFW now provides the \mbox{\hyperlink{window_guide_GLFW_WIN32_KEYBOARD_MENU_hint}{GLFW\+\_\+\+WIN32\+\_\+\+KEYBOARD\+\_\+\+MENU}} window hint for enabling keyboard access to the window menu via the Alt+\+Space and Alt-\/and-\/then-\/\+Space shortcuts. This may be useful for more GUI-\/oriented applications.\hypertarget{news_caveats}{}\doxysubsection{Caveats for version 3.\+4}\label{news_caveats}
\hypertarget{news_native_34}{}\doxysubsubsection{Multiple sets of native access functions}\label{news_native_34}
Because GLFW now supports runtime selection of platform (window system), a library binary may export native access functions for multiple platforms. Starting with version 3.\+4 you must not assume that GLFW is running on a platform just because it exports native access functions for it. After initialization, you can query the selected platform with glfw\+Get\+Platform.\hypertarget{news_version_string_34}{}\doxysubsubsection{Version string format has been changed}\label{news_version_string_34}
Because GLFW now supports runtime selection of platform (window system), the version string returned by glfw\+Get\+Version\+String has been expanded. It now contains the names of all APIs for all the platforms that the library binary supports.\hypertarget{news_joysticks_34}{}\doxysubsubsection{Joystick support is initialized on demand}\label{news_joysticks_34}
The joystick part of GLFW is now initialized when first used, primarily to work around faulty Windows drivers that cause Direct\+Input to take up to several seconds to enumerate devices.

This change will usually not be observable. However, if your application waits for events without having first called any joystick function or created any visible windows, the wait may never unblock as GLFW may not yet have subscribed to joystick related OS events.

To work around this, call any joystick function before waiting for events, for example by setting a \mbox{\hyperlink{input_guide_joystick_event}{joystick callback}}.\hypertarget{news_wayland_alpha_34}{}\doxysubsubsection{Frambuffer may lack alpha channel on older Wayland systems}\label{news_wayland_alpha_34}
On Wayland, when creating an EGL context on a machine lacking the new {\ttfamily EGL\+\_\+\+EXT\+\_\+present\+\_\+opaque} extension, the \mbox{\hyperlink{window_guide_GLFW_ALPHA_BITS}{GLFW\+\_\+\+ALPHA\+\_\+\+BITS}} window hint will be ignored and the framebuffer will have no alpha channel. This is because some Wayland compositors treat any buffer with an alpha channel as per-\/pixel transparent.

If you want a per-\/pixel transparent window, see the \mbox{\hyperlink{window_guide_GLFW_TRANSPARENT_FRAMEBUFFER_hint}{GLFW\+\_\+\+TRANSPARENT\+\_\+\+FRAMEBUFFER}} window hint.\hypertarget{news_standalone_34}{}\doxysubsubsection{Tests and examples are disabled when built as a subproject}\label{news_standalone_34}
GLFW now does not build the tests and examples when it is added as a subdirectory of another CMake project. To enable these, set the \mbox{\hyperlink{compile_guide_GLFW_BUILD_TESTS}{GLFW\+\_\+\+BUILD\+\_\+\+TESTS}} and \mbox{\hyperlink{compile_guide_GLFW_BUILD_EXAMPLES}{GLFW\+\_\+\+BUILD\+\_\+\+EXAMPLES}} cache variables before adding the GLFW subdirectory.

\hypertarget{news_initmenu_34}{}\doxysubsubsection{mac\+OS main menu now created at initialization}\label{news_initmenu_34}
GLFW now creates the main menu and completes the initialization of NSApplication during initialization. Programs that do not want a main menu can disable it with the \mbox{\hyperlink{intro_guide_GLFW_COCOA_MENUBAR_hint}{GLFW\+\_\+\+COCOA\+\_\+\+MENUBAR}} init hint.\hypertarget{news_corevideo_34}{}\doxysubsubsection{Core\+Video dependency has been removed}\label{news_corevideo_34}
GLFW no longer depends on the Core\+Video framework on mac\+OS and it no longer needs to be specified during compilation or linking.\hypertarget{news_caveat_fbtransparency_34}{}\doxysubsubsection{Framebuffer transparency requires DWM transparency}\label{news_caveat_fbtransparency_34}
GLFW no longer supports framebuffer transparency enabled via GLFW\+\_\+\+TRANSPARENT\+\_\+\+FRAMEBUFFER on Windows 7 if DWM transparency is off (the Transparency setting under Personalization \texorpdfstring{$>$}{>} Window Color).\hypertarget{news_emptyevents_34}{}\doxysubsubsection{Empty events on X11 no longer round-\/trip to server}\label{news_emptyevents_34}
Events posted with glfw\+Post\+Empty\+Event now use a separate unnamed pipe instead of sending an X11 client event to the helper window.\hypertarget{news_deprecations_34}{}\doxysubsection{Deprecations in version 3.\+4}\label{news_deprecations_34}

\hypertarget{news_vulkan_static_34}{}\doxysubsubsection{GLFW\+\_\+\+VULKAN\+\_\+\+STATIC CMake option has been removed}\label{news_vulkan_static_34}
This option was used to compile GLFW directly linked with the Vulkan loader, instead of using dynamic loading to get hold of {\ttfamily vk\+Get\+Instance\+Proc\+Addr} at initialization. This is now done by calling the glfw\+Init\+Vulkan\+Loader function before initialization.

If you need backward compatibility, this macro can still be defined for GLFW 3.\+4 and will have no effect. The call to glfw\+Init\+Vulkan\+Loader can be conditionally enabled in your code by checking the GLFW\+\_\+\+VERSION\+\_\+\+MAJOR and GLFW\+\_\+\+VERSION\+\_\+\+MINOR macros.\hypertarget{news_osmesa_option_34}{}\doxysubsubsection{GLFW\+\_\+\+USE\+\_\+\+OSMESA CMake option has been removed}\label{news_osmesa_option_34}
This option was used to compile GLFW for the Null platform. The Null platform is now always supported. To produce a library binary that only supports this platform, the way this CMake option used to do, you will instead need to disable the default platform for the target OS. This means setting the \mbox{\hyperlink{compile_guide_GLFW_BUILD_WIN32}{GLFW\+\_\+\+BUILD\+\_\+\+WIN32}}, \mbox{\hyperlink{compile_guide_GLFW_BUILD_COCOA}{GLFW\+\_\+\+BUILD\+\_\+\+COCOA}} or \mbox{\hyperlink{compile_guide_GLFW_BUILD_X11}{GLFW\+\_\+\+BUILD\+\_\+\+X11}} CMake option to false.

You can set all of them to false and the ones that don\textquotesingle{}t apply for the target OS will be ignored.\hypertarget{news_wl_shell_34}{}\doxysubsubsection{Support for the wl\+\_\+shell protocol has been removed}\label{news_wl_shell_34}
Support for the wl\+\_\+shell protocol has been removed and GLFW now only supports the XDG-\/\+Shell protocol. If your Wayland compositor does not support XDG-\/\+Shell then GLFW will fail to initialize.\hypertarget{news_symbols_34}{}\doxysubsection{New symbols in version 3.\+4}\label{news_symbols_34}
